# 面向 AI 驱动 UI 测试的 UX 启发式

这些启发式（heuristic）用于指导 LLM 在评估截图和页面交互时的判断。共五套评估框架，针对 SaaS 应用、仪表盘以及内部工具。

---

## 1. UX 定律（Laws of UX，行为与认知）

### 菲茨定律（Fitts's Law）
**原则：** 触达一个目标所需的时间是距离和大小的函数。
**检查要点：**
- 主要行动号召（CTA）应当是所在区块中最大的可点击元素
- 破坏性操作（删除、取消）的尺寸不应大于建设性操作
- 移动端点按目标至少 44x44px（Apple HIG）/ 48x48px（Material）
- 导航项应有充裕的内边距，而不仅仅是文字大小的命中区域
- 表单提交按钮在移动端应为全宽
- 高频操作应更靠近用户当前的焦点区域

### 希克定律（Hick's Law）
**原则：** 决策时间随选项的数量与复杂度增加而增加。
**检查要点：**
- 选项超过 7±2 个的导航菜单应分组/分类
- 选项众多的设置页应使用分区/标签页，而非平铺的列表
- 弹窗应呈现一个清晰的操作，而非多个相互竞争的选择
- 含 15 个以上条目的下拉菜单应提供搜索/筛选
- 引导流程不应一次性呈现所有选项
- 操作菜单（右键、"更多"菜单）应按使用频率组织

### 米勒定律（Miller's Law）
**原则：** 工作记忆一次约能保留 7±2 个项目。
**检查要点：**
- 长列表应分块为 5-9 项一组
- 电话号码、信用卡号和验证码应做视觉分组
- 仪表盘应优先呈现 3-5 个关键指标，而非全部显示
- 侧边栏导航应将相关项归入标题下分组
- 多步骤向导显示的步骤不应超过 5-7 步

### 雅各布定律（Jakob's Law）
**原则：** 用户的大部分时间花在其他网站上，并期望你的网站以同样的方式运作。
**检查要点：**
- 登录表单应将邮箱/用户名置于上方、密码在下方、提交在底部
- Logo 应链接到首页/仪表盘
- 外部链接应在视觉上有所区分，或在新标签页打开
- 返回按钮应在左上角
- 搜索应在头部区域
- 设置齿轮图标应通向设置
- 用户头像/菜单应在右上角
- 侧边栏导航在左，内容在右

### 美观可用性效应（Aesthetic-Usability Effect）
**原则：** 用户会把美观的设计感知为更易用。
**检查要点：**
- 所有组件的圆角保持一致
- 统一的配色（主色不超过 3-4 种 + 中性色）
- 一致的间距节奏（4px 或 8px 的倍数）
- 排版层次清晰（标题 > 副标题 > 正文 > 说明文字）
- 对齐方式一致（文本左对齐、标题居中等）
- 没有缺乏视觉分组、孤零零漂浮的元素

### 多尔蒂阈值（Doherty Threshold）
**原则：** 当系统响应 <400ms 时，生产力会大幅提升。
**检查要点：**
- 用户操作的系统响应时间 < 400ms
- 超过 1 秒的操作出现加载指示
- 常见操作（开关、点赞、保存）采用乐观 UI 更新（optimistic UI）
- 内容加载使用骨架屏，而不仅仅是转圈
- 页内操作不触发整页刷新

### 冯·雷斯托夫效应（Von Restorff Effect，隔离效应）
**原则：** 从同类中脱颖而出的项目更令人难忘。
**检查要点：**
- 每页最重要的元素在视觉上与众不同
- 定价页突出推荐的方案
- 错误提示从周围内容中凸显出来
- 新增/更新功能有视觉标识（徽标、高亮）
- 主要操作与次要操作在视觉上有所区分

### 蔡格尼克效应（Zeigarnik Effect）
**原则：** 人们对未完成的任务比已完成的任务记得更牢。
**检查要点：**
- 多步骤表单显示进度指示
- 引导流程显示完成百分比
- 如相关，则显示资料/设置的完成度
- 未完成任务在视觉上与已完成任务有所区别

## 2. 尼尔森十大可用性启发式（Nielsen's 10 Usability Heuristics）

### H1：系统状态的可见性（Visibility of System Status）
**原则：** 系统应始终让用户了解正在发生什么。
**检查要点：**
- 当前页面/区块在导航中被高亮
- 面包屑显示用户在层级中的位置
- 表单提交显示成功/失败反馈
- 文件上传显示带百分比的进度条
- 后台操作显示状态（同步中、保存中、处理中）
- 生效的筛选/排序有视觉指示
- 多步骤流程中存在"你在这里"的指示
- 时间戳显示数据上次更新时间（"2 分钟前更新"）

### H2：系统与真实世界的匹配（Match Between System and Real World）
**原则：** 使用用户熟悉的语言和概念，而非面向系统的术语。
**检查要点：**
- 错误提示使用通俗语言，而非错误码或堆栈信息
- 标签使用用户理解的领域术语
- 图标契合真实世界的隐喻（垃圾桶表示删除等）
- 日期使用用户所在地区的格式，而非 ISO/Unix 时间戳
- 数字使用恰当的格式（千分位、货币符号、缩写）
- 状态标签有实际意义（"正在处理付款"，而非"State: 3"）

### H3：用户的控制权与自由度（User Control and Freedom）
**原则：** 用户需要一个清晰标记的"紧急出口"来脱离不想要的状态。
**检查要点：**
- 破坏性操作可撤销
- 所有表单和弹窗都有取消按钮
- 返回导航有效且保留状态
- 多步骤流程允许返回上一步
- 批量选择可一键清除
- 筛选条件可重置为默认值
- 弹窗/对话框可通过 Escape 键以及点击外部关闭

### H4：一致性与标准（Consistency and Standards）
**原则：** 用户不应去琢磨不同的词语、情境或操作是否表示同一件事。
**检查要点：**
- 同一操作在各处使用相同的按钮样式（主要、次要、破坏性）
- 术语保持一致（不要对同一操作混用"删除"和"移除"）
- 图标在整个应用中含义相同
- 日期/时间格式自始至终一致
- 表格列的行为保持一致（要么都可排序，要么都不可）
- 各区块的空状态遵循相同的模式
- 错误提示样式一致（toast、内联、横幅）

### H5：错误预防（Error Prevention）
**原则：** 比好的错误提示更好的，是从一开始就预防错误。
**检查要点：**
- 破坏性操作需要确认（"确定要删除吗？"）
- 表单输入有恰当的约束（type=email、maxlength、pattern）
- 禁用状态阻止不可能的操作（不能提交空表单）
- 危险按钮在视觉上有所区分（红色），且不与安全按钮相邻
- 未保存的更改触发"离开页面？"确认
- 可用时指示自动保存
- 输入框在用户输入前显示格式提示（占位符、辅助文本）

### H6：识别优于回忆（Recognition Rather Than Recall）
**原则：** 通过让对象、操作和选项可见来最小化记忆负荷。
**检查要点：**
- 显示最近使用过的项目（最近搜索、最近文件）
- 表单字段显示当前值，而非需要用户记忆的空字段
- 导航标签可见，而非仅藏在图标背后
- 搜索带自动补全/建议
- 在需要处提供上下文帮助（工具提示、信息图标）
- 仪表盘小组件显示标签，而非只有数字

### H7：使用的灵活性与效率（Flexibility and Efficiency of Use）
**原则：** 面向专家用户的加速器不应妨碍新手用户。
**检查要点：**
- 为高级用户提供键盘快捷键（并且可被发现）
- 为重复任务提供批量操作
- 长列表提供搜索/筛选
- 为 ID、密钥、URL 提供一键复制
- 提供快捷操作（内联编辑、单击操作）
- 默认值合理且能节省时间

### H8：美观与极简设计（Aesthetic and Minimalist Design）
**原则：** 每一份多余的信息都会与相关信息争夺注意力。
**检查要点：**
- 页面上没有冗余信息
- 首屏内容是最重要的内容
- 视觉噪音被最小化（不必要的边框、分隔线、装饰）
- 用留白来制造焦点
- 次要信息可访问但不显眼（可展开区块、工具提示）
- 表单只索取真正需要的内容

### H9：帮助用户识别、诊断并从错误中恢复（Help Users Recognize, Diagnose, and Recover from Errors）
**原则：** 错误提示应以通俗语言表达，指出问题，并给出解决方案。
**检查要点：**
- 错误提示以通俗语言解释**出了什么问题**
- 错误提示建议**如何**修复
- 表单校验错误出现在相关字段旁，而非只在顶部
- 错误状态不丢失用户的输入（出错后保留表单数据）
- 网络错误提供重试选项
- 404 页面给出替代方案（搜索、导航、首页链接）
- API 错误不向用户泄露技术细节（无堆栈信息、无原始 JSON）

### H10：帮助与文档（Help and Documentation）
**原则：** 帮助应易于检索、聚焦用户任务且简洁。
**检查要点：**
- 复杂功能有上下文帮助（工具提示、"?"图标、内联提示）
- 为新用户提供引导/导览
- 空状态包含关于下一步该做什么的指引
- 可从应用内访问文档链接
- 恰当时，错误状态链接到相关帮助文章

## 3. 错误状态与边界情况

SaaS 应用 80% 的开发时间花在主流程（happy path）上，20% 花在其他所有情况上。这些检查捕捉来自"其他所有情况"的那 80% 的用户挫败感。

### 空状态
**检查要点：**
- 每个列表/表格都有专门设计的空状态（而非一片空白）
- 空状态解释这个区块是做什么的
- 空状态包含一个创建首个项目的 CTA
- 空状态有插图或图标（而非只有文字）
- 无搜索结果时提供建议或"换个关键词试试"
- 筛选后无匹配的视图显示"没有结果符合你的筛选条件"，并附清除筛选按钮

### 错误边界
**检查要点：**
- JavaScript 错误不会让整个页面崩溃（错误边界将其捕获）
- 失败的 API 调用显示有意义的错误，而非空白区域
- 网络断开显示离线指示
- 超时错误提供重试
- 认证过期重定向到登录页并附说明，而非显示一个损坏的页面
- 部分数据加载不会让 UI 停留在渲染了一半的状态

### 表单边界情况
**检查要点：**
- 极长的文本输入不破坏布局（截断或滚动）
- 输入中的特殊字符不引发错误（引号、unicode、emoji）
- 把内容粘贴到字段中能正确工作
- 双击提交不会创建重复条目
- 必填字段标识在提交前就可见，而非提交后才显示
- 表单字段的 Tab 顺序遵循视觉顺序
- 日期选择器能处理时区边界情况
- 数字输入能处理负数、小数和零

### 加载状态
**检查要点：**
- 初次页面加载有骨架屏或加载指示
- 数据刷新显示不显眼的加载指示（而非整页转圈）
- 长时间运行的操作显示进度（而非只有转圈）
- 指示过期数据（"5 分钟前更新"或视觉变暗）
- 若服务器拒绝，乐观更新能优雅回退
- 无限滚动在底部有加载指示
- 图片加载使用模糊渐显、骨架或占位符

### 权限与认证边界情况
**检查要点：**
- 未授权访问显示有意义的提示，而非 403 页面
- 会话过期重定向到登录页，并将用户带回原来的位置
- 基于角色的 UI 隐藏用户无法执行的操作（而非仅禁用）
- 共享链接对有相应权限的用户有效
- "拒绝访问"提示建议联系谁以获取权限

## 4. 数据展示启发式

SaaS 应用数据密集。表格、图表、仪表盘和列表是用户花费大部分时间的地方，也是视觉缺陷和可用性问题最常见的来源。

### 表格
**检查要点：**
- 列标题清晰简洁
- 列宽恰当（并非全都等宽）
- 长单元格内容用工具提示/展开截断，而非溢出
- 数值列右对齐
- 文本列左对齐
- 存在行悬停状态以便视觉追踪
- 可排序列有指示（排序图标）
- 显示当前排序方向（升序/降序）
- 空表格有专门设计的空状态
- 分页显示总数和当前范围（"显示第 1-25 条，共 142 条"）
- 表格可用键盘导航
- 移动端：表格横向滚动 或 转为卡片布局

### 图表
**检查要点：**
- 图表有清晰的标题和坐标轴标签
- 图例可见且与图表颜色匹配
- 悬停时工具提示显示精确数值
- 无数据状态显示一条提示，而非空图表
- 配色无障碍（不仅依赖颜色来区分数据系列）
- 柱状图 Y 轴从零开始（除非有明确理由不这么做）
- 时间序列 X 轴有恰当的间隔
- 图表是响应式的（在小屏上不溢出）

### 仪表盘指标
**检查要点：**
- 关键指标有标签、数值，以及上下文（趋势、对比周期）
- 大数字使用恰当格式（1.2K 而非 1200，$1.5M 而非 $1500000）
- 百分比变化显示方向（上/下箭头或颜色）
- 指示指标的刷新（时间戳或刷新按钮）
- 指标卡片尺寸和对齐一致
- 指标过多会造成认知超载 —— 优先呈现 3-5 个关键指标

### 筛选与搜索
**检查要点：**
- 生效的筛选条件可见且可逐个移除
- 任一筛选条件生效时存在"清除所有筛选"按钮
- 筛选状态在页面导航间保持
- 搜索具响应性（边输入边出结果，或按回车后出结果）
- 搜索能优雅处理空查询
- 产生零结果的筛选组合显示有帮助的提示
- 日期范围筛选校验起始 < 结束

### 分页与无限滚动
**检查要点：**
- 显示项目总数
- 指示当前页/位置
- 表格存在每页条数选择器
- URL 随页面状态更新（可分享/可收藏）
- 返回时保留滚动位置和页码
- 无限滚动有明确的"列表到底"指示
- 加载更多项目不会跳动滚动位置

### 数字、日期与格式
**检查要点：**
- 日期在整个应用中使用一致的格式
- 恰当处使用相对日期（"3 小时前"而非"2026-03-24T16:30:00Z"）
- 货币值显示恰当的符号和小数位
- 大数字以一致的方式缩写（K、M、B）
- 百分比显示恰当的精度（不要显示 33.333333%）
- 时长格式可读（"2h 15m"而非"135 minutes"或"8100000ms"）
- 空值/未定义值显示破折号或"N/A"，而非"null""undefined"或空白

## 5. 视觉设计检查

### 排版
- 正文：桌面端最小 16px，移动端最小 14px
- 行高：正文 1.4-1.6
- 标题刻度：h1 → h2 → h3 间有清晰的大小差异
- 最大行长：60-80 个字符以保证可读性
- 一个页面上字体族不超过 2-3 种
- 字重变化是有意为之且一致的
- 代码、ID 和技术值使用等宽字体

### 颜色与对比度
- WCAG AA：普通文本对比度 4.5:1
- WCAG AA：大号文本（18px+ 或 14px+ 加粗）对比度 3:1
- 交互元素有明显的悬停/激活/聚焦状态
- 错误状态使用红色（或符合文化习惯的警示色）
- 成功状态使用绿色
- 警告状态使用黄色/琥珀色
- 信息状态使用蓝色
- 禁用状态视觉上变弱但仍可读
- 状态颜色在整个应用中一致（同样的绿色 = 同样的含义）

### 间距与布局
- 网格列之间的间距一致
- 区块间距遵循可预测的节奏
- 相关项目比无关项目挨得更近（邻近原则）
- 留白是有意为之，而非偶然
- 没有内容在没有内边距的情况下紧贴视口（viewport）边缘
- 卡片/容器有一致的内部内边距
- 侧边栏宽度恰当（不至于窄到难读，也不至于宽到挤占内容）

### 交互元素
- 所有按钮有可见的悬停状态
- 所有链接可与普通文本区分
- 表单输入有清晰的聚焦状态（而非仅浏览器默认轮廓）
- 禁用元素看起来就是禁用的（变弱、无指针光标）
- 加载中的按钮显示转圈并阻止双击
- 开关/切换的开与关状态清晰可辨
- 下拉指示（折叠箭头）方向正确（收起时朝下，展开时朝上）
- 破坏性按钮在视觉上有所区分（红色或描边，而非主要样式）

## 6. 无障碍检查

### 键盘导航
- 所有交互元素可通过 Tab 触达
- Tab 顺序遵循视觉顺序（从上到下、从左到右）
- 所有元素上聚焦可见（而非仅浏览器默认 —— 自定义聚焦环）
- Escape 关闭弹窗/下拉
- Enter/Space 激活按钮
- 方向键在组件组内导航（标签页、菜单、单选组）
- 为屏幕阅读器用户提供跳至内容（skip-to-content）链接
- 焦点被困在打开的弹窗内（无法 Tab 到弹窗背后的元素）
- 弹窗关闭时焦点返回触发元素

### 屏幕阅读器
- 所有图片有 alt 文本（装饰性图片则用 aria-hidden）
- 表单输入有关联的标签（而非仅占位符文本）
- 标题层级化（不要从 h1 跳到 h4）
- 存在地标区域（nav、main、aside、footer）
- 实时区域（live region）播报动态内容更新（toast 通知、加载状态）
- 弹窗困住焦点并播报其标题
- 表格有恰当的 th/td 结构和 scope 属性
- 自定义组件（标签页、手风琴、下拉）有正确的 ARIA 角色

### 动效与减弱动效
- 遵从 `prefers-reduced-motion` 媒体查询
- 没有无法暂停的自动播放动画
- 没有闪烁内容（每秒最多 3 次闪烁）
- 视差/滚动触发的动画有替代方案
- 加载转圈简单（无复杂动画）

### 颜色独立性
- 信息绝不仅靠颜色传达
- 错误状态同时有图标和颜色（而非仅红色文字）
- 图表在颜色之外还使用图案/形状
- 状态指示在颜色圆点旁有文字标签
- 表单校验显示图标（对勾、叉）而非仅绿/红边框
